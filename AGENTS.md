# AI Agent Instructions

## Project Overview

Email Processing Service — a hybrid (AWS + self-hosted) platform that receives emails via SES, classifies them using an LLM, and routes them: supplier files are stored, deal questions trigger admin notifications.

## Architecture

### Backend Services

| Service | Responsibility |
|---|---|
| ConfigService | Zod-validated env loading |
| StoreService | Drizzle ORM CRUD for emails, supplierFiles, dealQuestions |
| AuthService | JWT sign/verify, in-memory refresh tokens |
| EmailParserService | Parse raw MIME emails from S3 |
| EmailProcessingService | Orchestrator: coordinates detection → storage → notification |
| QuestionClassifierService | LLM call (OpenRouter) to classify deal questions |
| SupplierDetectionService | LLM call (OpenRouter) to detect supplier files + map columns |
| S3Service / SesService / SqsService | AWS SDK v3 wrappers |
| NotificationService | Send admin email via SES |
| PollerService | Background SQS polling loop |

### Database (3 tables)

- **emails** — raw email metadata, processing status, S3 key, attachments (JSONB)
- **supplierFiles** — detected supplier XLSX files, column mapping (JSONB), preview rows
- **dealQuestions** — classified questions, notification status

Relations: emails → supplierFiles (1:N), emails → dealQuestions (1:N)

### AWS Resources (Terraform)

SES (receive email) → S3 (store raw) + SQS (notify) → Backend polls SQS → processes email

## Coding Conventions

### General
- TypeScript strict mode everywhere
- camelCase for variables/functions, PascalCase for classes/interfaces/types
- Async/await, never raw `.then()` chains
- Explicit return types on all exported functions and methods

### Backend (NestJS)
- Constructor-based dependency injection
- Single `AppModule` — no sub-modules
- Class-based middleware implementing `NestMiddleware`
- Use NestJS `Logger` (not `console.log`)
- Throw NestJS HTTP exceptions (`UnauthorizedException`, `HttpException`)
- Drizzle typed queries with builder pattern: `db.select().from(table).where()`
- Config accessed via injected `ConfigService`, never `process.env` directly

### Frontend (React)
- Functional components with hooks only (no class components)
- All API calls go through `api.ts` centralized client
- JWT tokens stored in localStorage (`accessToken`, `refreshToken`)
- TailwindCSS utility classes for styling, no CSS modules
- Zod schema for env validation in `vite.config.ts`

### Database
- Schema defined in `backend/src/services/db/schema.ts`
- Migrations auto-generated by Drizzle Kit — never edit SQL files in `drizzle/` manually
- Use JSONB for flexible/nested data (attachments, column mappings, preview rows)
- UUID primary keys, timestamp columns as ISO strings

### Infrastructure
- All AWS resources provisioned via Terraform in `terraform/`
- Variables prefixed with `TF_VAR_` in `.env`
- Use `scripts/load-terraform-env.sh` before `terraform plan/apply`

## Key Design Decisions

- **Hybrid architecture**: AWS handles email ingestion (SES → S3 → SQS), backend runs self-hosted and polls SQS
- **LLM classification via OpenRouter**: emails are classified as supplier files or deal questions using `openrouter/auto` model
- **Idempotency**: `messageId` is unique on the emails table to prevent duplicate processing
- **Column mapping**: supplier file columns are mapped to expected schema via LLM, stored as JSONB
- **Auth**: simple JWT + refresh token flow, refresh tokens stored in-memory (not persisted)
